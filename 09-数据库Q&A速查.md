# 数据库知识点Q&A速查

> 本文档整理了数据库学习中的常见问题与解答，按照学习顺序排列，适合复习和面试备考。

## 一、数据库概述与发展

### Q1：数据库系统有哪些关键概念？

**核心术语**：

| 术语 | 全称 | 说明 |
|------|------|------|
| **DB** | Data Base | 以一定组织形式存储的数据集合 |
| **DBMS** | Data Base Management System | 管理和维护数据库的系统软件 |
| **DBS** | Data Base System | 包含DB、DBMS、DBA的完整系统 |

**DBA职责（必背）**：
1. 决定数据库中的信息内容和结构
2. 决定数据库的存储结构和存取策略
3. 定义数据安全性和完整性约束
4. 监控数据库的使用和运行
5. 负责数据库的改进和重组

## 二、DBMS内部结构

### Q2：先写日志原则(WAL)是什么？

**一句话核心**：先落盘日志，再落盘真实数据，专门防"断电/崩溃导致数据残缺、不一致"。

**为什么必须这么做？**
- 如果先写数据、后写日志：数据已写到磁盘，但日志还没写
- 中途宕机后：没有日志记录，无法回滚/校验，直接数据错乱
- 遵循WAL：日志完整，重启靠日志就能恢复

**标准WAL流程**：
```
1. 执行SQL修改数据
2. 优先把redo日志写入磁盘（先落盘）
3. 修改内存中的数据页
4. 后台异步把内存数据刷到磁盘
5. 数据正常落盘后，对应日志可清理
```

---

### Q3：日志记录格式是干嘛的？

**一句话总结**：日志记录格式 = 数据库日志的"统一模板/固定写法"，规定每条日志必须包含的内容。

**为什么需要格式？**
- 如果没有固定格式，日志会变乱：有的只写值，有的写一半
- 崩溃恢复时，数据库看不懂日志 → 无法回滚/重做 → 数据错乱或丢失

**三种日志格式对比**：

| 日志类型 | 记录内容 | 作用 |
|----------|----------|------|
| **undo日志** | 旧值 | 回滚，恢复修改前的值 |
| **redo日志** | 新值 | 重做，补写修改后的值 |
| **undo/redo日志** | 旧值+新值 | 既能回滚，又能重做 |

---

### Q4：redo日志是干嘛用的？

**核心用途**：解决"数据库宕机，内存数据没刷到磁盘"的数据丢失问题。

**工作场景**：
1. 数据库修改数据时：先写redo日志（顺序写，快）→ 再改内存数据 → 后台慢慢刷到磁盘
2. 如果还没刷盘就断电：内存数据直接清空，磁盘还是旧数据
3. 重启后扫描redo日志，按日志内容重新执行修改操作

**一句话区分**：
- **undo**：事务中途失败 → 回滚，恢复旧值
- **redo**：事务已提交、但数据未落盘 → 重做，补写新值

---

### Q5：redo这么能存还要磁盘干嘛？

**核心结论**：redo日志不能替代数据磁盘，二者分工完全不同。

| 对比项 | 数据磁盘 | redo日志 |
|--------|----------|----------|
| 本质 | 存最终生效的业务数据 | 只存数据变更记录 |
| 作用 | 对外提供读写 | 崩溃后重做恢复 |
| 读写特点 | 随机读写 | 顺序写入 |
| 生命周期 | 永久保存 | 短期留存，落盘后可清理 |

---

### Q6：检查点(Checkpoint)的作用是什么？

**一句话核心**：检查点 = 数据库的"存档点"，把内存里没写到硬盘的数据一次性刷进硬盘，方便崩溃后快速恢复。

**三大作用**：
1. **加速崩溃恢复**：只恢复检查点之后的日志
2. **释放日志空间**：旧日志可以覆盖
3. **减少内存压力**：把数据刷回硬盘

**和WAL的区别**：

| 概念 | 定位 |
|------|------|
| **WAL（先写日志）** | 运行规则，保证能恢复 |
| **检查点** | 性能优化，让恢复变快 |

---

### Q7：ACID中的C是什么意思？

**ACID中的C = Consistency（一致性）**

**定义**：事务执行前后，数据库整体数据必须保持合法、符合预设规则，不会出现逻辑错乱。

**经典例子（转账）**：
- 规则：总金额不变
- 初始：A=200，B=200，总额400（合法）
- 中间临时：A=100，B=200（总额300，临时不一致，允许）
- 事务完成：A=100，B=300，总额400（恢复一致）

**易混区分**：
- C一致性：管数据逻辑、业务规则
- I隔离性：管多个事务并发互相干扰
- D持久性：管提交后数据不丢失

---

### Q8：故障类型与恢复方法

| 故障类型 | 恢复方法 | 为什么 |
|----------|----------|--------|
| **事务故障** | UNDO（回滚） | 单事务失败，日志留存，需撤销修改 |
| **系统故障** | UNDO + REDO | 既有未提交事务，也有已提交未落盘事务 |
| **介质故障** | 备份 + REDO | 备份+归档日志仅保留已提交数据，无未提交事务 |

## 三、三级映射模式

### Q9：什么是逻辑独立性？

| 项目 | 说明 |
|------|------|
| **定义** | 概念模式改变时，外模式不变 |
| **原因** | 外模式/概念模式映射 |
| **影响** | 应用程序不需要改变 |

**举例**：
- 增加新字段 → 概念模式变，外模式不变
- 修改字段名 → 修改映射即可

---

### Q10：什么是物理独立性/物理数据独立性？

**定义**：内模式改变时，概念模式不变。

**触发场景**：
- 改变存储结构
- 建立新索引
- 把数据挪到另一块磁盘

**一句话区分**：
- 逻辑独立性：表结构/字段变了（逻辑变）→ 程序不动
- 物理独立性：存储/索引/磁盘变了（物理变）→ 程序不动

## 四、ER模型

### Q11：什么是弱实体集？

**一句话核心**：弱实体集 = 自己活不了，必须依附别人才能活的实体。

**弱实体集特征（必背）**：
1. 没有自己的主键
2. 必须依赖强实体存在
3. 画ER图时：弱实体用**双矩形**，依赖联系用**双菱形**
4. 主键 = 强实体主键 + 弱实体的分辨符

**经典例子**：
- 强实体：部门（Department），主键DNo
- 弱实体：课程（Course），分辨符CNo
- 必须用 **DNo + CNo** 才能唯一标识一门课

**速记**：
- 强实体：我自己就能活
- 弱实体：没爹我活不了

## 五、数据定义

### Q12：完整性约束包括哪些？

数据库完整性约束分为**四大类**：

| 约束类型 | 约束对象 | 规则 | 语法 |
|----------|----------|------|------|
| **实体完整性** | 表中的每一条记录 | 主键非空、唯一 | PRIMARY KEY |
| **域完整性** | 字段本身的值 | 数据类型、非空、默认值 | NOT NULL, DEFAULT |
| **参照完整性** | 多表之间的关联 | 外键取值必须匹配主表主键 | FOREIGN KEY |
| **用户自定义完整性** | 业务专属规则 | 业务逻辑约束 | CHECK |

**速记口诀**：
- 实体完整性 → 主键（唯一、非空）
- 域完整性 → 列（类型、非空、默认）
- 参照完整性 → 外键（表与表关联）
- 用户自定义完整性 → 业务专属规则

## 六、各种数据库

### Q13：MongoDB是什么？

**一句话总结**：MongoDB = 灵活的JSON数据库 + 天生分布式 + 高并发读写。

**和MySQL最大的不同**：

| MongoDB | MySQL |
|---------|-------|
| 不用预先定表结构 | 先建表、定死字段 |
| 存成BSON文档 | 行列严格的表格 |
| 字段随便加、随便改 | 改表麻烦 |

**核心概念对照**：

| MongoDB | MySQL | 说明 |
|---------|-------|------|
| Database | Database | 数据库实例 |
| Collection | Table | 集合/表 |
| Document | Row | 文档/行 |
| Field | Column | 字段 |

**适合场景**：
- 数据结构经常变（敏捷开发）
- 半结构化/JSON数据
- 高并发读写、海量数据

**不适合**：
- 强事务、强一致性（金融核心交易）
- 复杂多表JOIN报表

## 七、数据库设计

### Q14：BCNF和第二范式为什么这么像？

**核心区别**：

| 对比项 | 2NF | BCNF |
|--------|-----|------|
| **约束范围** | 只管**非主属性** | 管**所有属性** |
| **禁止问题** | 非主属性部分依赖候选键 | 任何依赖左部不是候选键 |
| **主属性依赖** | 完全不管 | 零容忍 |

**经典反例（满足2NF/3NF但违反BCNF）**：
```
课程安排(教师, 课程, 教室)
假设：教师→课程，课程→教室
分析：
- 所有属性全是主属性 → 满足2NF、3NF
- 但教师→课程，左部教师不是候选键 → 违反BCNF
```

**速记口诀**：
- 2NF：别让普通字段蹭主键的一部分（禁止非主属性部分依赖）
- BCNF：任何字段，都不能被不是候选键的字段决定（全员约束）

## 八、事务与并发控制

### Q15：什么是不可重复读？

**定义**：同一个事务内，对同一条数据先后读两次，两次读到的结果不一样。

**例子**：
```
T1: 第一次读 A=100
T2: 修改A=200，提交
T1: 第二次读 A=200 → 两次结果不同 = 不可重复读
```

**和脏读的区别**：
- **脏读**：读到未提交、后续被回滚的临时数据
- **不可重复读**：同一事务多次读，结果不一致（数据被别人提交修改）

---

### Q16：什么是幻读？

**定义**：同一事务内，按条件多次查询，前后查到的行数/记录不一样，像凭空出现的"幻影"。

**例子**：
```
T1: SELECT * FROM 学生 → 查到2条
T2: 插入新学生，提交
T1: 再次SELECT → 查到3条 → 幻读
```

**和不可重复读的核心区分**：

| 问题 | 什么变了 |
|------|----------|
| **不可重复读** | 同一行内容被修改（值变了，行数不变） |
| **幻读** | 新增/删除整行（行数变了） |

**速记**：
- 改内容 → 不可重复读
- 增减行数 → 幻读

---

### Q17：串行化和幻读是什么关系？

**核心结论**：串行化 = 唯一能100%彻底干掉幻读的隔离级别！

**为什么**：
- 串行化 = 让所有事务排队执行，不能同时跑
- 只要T1还没结束，别人根本不能插入新数据

**速记**：
- 可重复读：防不住幻读（别人还能偷偷插数据）
- 串行化：排队执行 → 幻读直接消失

---

### Q18：改数据和插数据有什么区别？

**核心区分**：

| 操作 | 导致的问题 |
|------|------------|
| **改数据（UPDATE）** | 不可重复读 |
| **插数据/删数据（INSERT/DELETE）** | 幻读 |

**对应隔离级别**：
- 可重复读 → 防修改（UPDATE）
- 串行化 → 防修改 + 防插入（INSERT/DELETE）

---

### Q19：封锁协议（一级、二级、三级）是什么？

**封锁协议 = 规定"什么时候加锁、什么时候释放锁"的规则**

| 协议 | 规则 | 解决问题 |
|------|------|----------|
| **一级** | 改前加X锁，事务结束才释放；读不加锁 | 丢失修改 |
| **二级** | 读前加S锁，读完即释放；其他同一级 | 丢失修改、脏读 |
| **三级** | 读前加S锁，事务结束才释放；其他同一级 | 全部三类问题 |

**为什么要给读专门加锁？**
- X锁只限制写操作，不限制原生的无锁读
- 光靠写锁挡不住脏读、不可重复读
- 给读加S锁，才能约束读行为

---

### Q20：什么是两段锁协议(2PL)？

**一句话核心**：前半段只管加锁、绝不解锁；后半段只管解锁、绝不加锁，两段不能交叉。

**正确示例**：
```
加锁X → 加锁Y → 解锁X → 解锁Y  ✅
```

**错误示例**：
```
加锁X → 解锁X → 加锁Y  ❌（穿插，违规）
```

**核心作用**：遵循两段锁协议 → 并发调度一定是串行化的

**针对单个事务**：每个事务都必须遵守两段锁规则，事务之间互不干涉。

---

### Q21：封锁粒度和意向锁是什么？

**封锁粒度 = 你锁多大的东西**

| 粒度 | 说明 |
|------|------|
| 属性/元组 | 锁小，并发高，开销大 |
| 页/块 | 平衡 |
| 表/数据库 | 锁大，并发低，开销小 |

**意向锁 = 给"大对象"打标记**

| 意向锁 | 含义 |
|--------|------|
| IS | 准备获取S锁（准备读行） |
| IX | 准备获取X锁（准备改行） |
| SIX | 持有表S锁，同时准备改某些行 |

**为什么需要意向锁**：
- 没有：加表锁前必须逐行检查有没有行锁，很慢
- 有：只要看表上有IS/IX标签，一眼判断，O(n)→O(1)

**意向锁之间不冲突**：
- IS、IX之间随便加，不冲突
- 只有意向锁和真正表锁（S/X）才冲突

---

### Q22：S锁和X锁的相容矩阵是什么？

| 已有锁 \ 申请锁 | 申请S锁 | 申请X锁 |
|-----------------|---------|---------|
| **已有S锁** | ✅ 允许（读读兼容） | ❌ 拒绝 |
| **已有X锁** | ❌ 拒绝 | ❌ 拒绝 |

**速记口诀**：
- 读读兼容：有S锁，还能再加S锁
- 读写互斥：有S锁，就加不了X锁
- 写写互斥：有X锁，啥锁都加不了

---

### Q23：隔离级别是什么？

**四个级别从松到严**：

| 隔离级别 | 解决了什么 | 还剩什么问题 |
|----------|------------|--------------|
| **未提交读** | 无 | 脏读、不可重复读、幻读 |
| **已提交读** | 脏读 | 不可重复读、幻读 |
| **可重复读** | 脏读、不可重复读 | 幻读 |
| **串行化** | 全部 | 无 |

**速记**：
- 未提交读：啥都不解决
- 已提交读：解决脏读
- 可重复读：解决脏读+不可重复读（MySQL默认）
- 串行化：全部解决

---

### Q24：一个事务可以同时申请多个读锁吗？

**可以！** 一个事务完全能同时申请多个S锁。

**原因**：S锁是共享锁，大家都能读，互相不冲突。

**对比**：
- 不同数据：可以加多把S锁、多把X锁
- 同一数据：不能同时加S和X锁（逻辑冲突）

---

### Q25：为什么一个事务不能对同一个数据同时加S和X锁？

**因为逻辑上完全矛盾**：
- S锁 = 只允许读，不允许改
- X锁 = 要读写独占

**数据库直接禁止**，同一数据上不能既是"只读"又是"可写独占"。

---

### Q26：锁兼容矩阵是给谁看的？

**锁兼容矩阵是给【不同事务之间】看的！**

| 已有锁 \ 申请锁 | S锁 | X锁 |
|-----------------|-----|-----|
| **已有S锁** | ✅ 别人能读 | ❌ 别人不能写 |
| **已有X锁** | ❌ 别人不能读 | ❌ 别人不能写 |

**重点**：所有✅❌都是在说【事务A】和【事务B】之间的关系，不是说"自己对自己"。

### 故障恢复对照

| 故障类型 | 恢复方法 | 核心特点 |
|----------|----------|----------|
| 事务故障 | UNDO | 单事务挂了，服务正常 |
| 系统故障 | UNDO + REDO | 整机宕机，磁盘完好 |
| 介质故障 | 备份 + REDO | 磁盘损坏，数据丢失 |

### 封锁协议对比

| 协议 | X锁规则 | S锁规则 | 解决问题 |
|------|---------|---------|----------|
| 一级 | 改前加，事务结束释放 | 不加 | 丢失修改 |
| 二级 | 同上 | 读前加，读完即放 | +脏读 |
| 三级 | 同上 | 读前加，事务结束放 | 全部解决 |

### 隔离级别对照

| 级别 | 脏读 | 不可重复读 | 幻读 |
|------|------|------------|------|
| 未提交读 | 可能 | 可能 | 可能 |
| 已提交读 | 不可能 | 可能 | 可能 |
| 可重复读 | 不可能 | 不可能 | 可能 |
| 串行化 | 不可能 | 不可能 | 不可能 |
